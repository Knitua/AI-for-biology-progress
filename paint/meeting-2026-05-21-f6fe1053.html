核心结论

  1. 直接 embedding 全部物种和候选基因，暂时不能支撑明确的蝙蝠特异性结论。 当前结果中，免疫相关基因在不同物种里本来就可能和其他基因分开；这种分离更像功能类别差异，而不一定是蝙蝠相对其他哺乳动物的特异性。把大量蝙蝠、对照物种和多个蛋白混在同一张图里，也会掩盖单一物种或单一蛋白的局部趋势。
  2. 参考文献中的进化树/同源基因分支分析仍是更稳妥的主线。 会议讨论了先识别同源基因、构建物种进化树，再沿祖先到子节点的分支观察基因或蛋白变化，并对突变较集中的基因做功能注释的路线。该流程更接近比较基因组领域可接受的分析范式，也更容易把结果连接到免疫通路和病毒耐受机制。
  3. 当前复现效果不佳，主要瓶颈可能在同源基因规模、建树规模和流程简化造成的误差积累。 已尝试的小规模树、较少同源基因和简化比对流程，未能稳定复现参考文献中免疫相关基因富集的效果。后续不应急于基于小规模结果下结论，而应优先把参考流程做完整、做大、做可复核。
  4. 数据优势集中在蝙蝠物种和高质量注释子集上，需要先把数据边界讲清楚。 会议中提到已下载约 28 个有基因/蛋白注释的蝙蝠物种，以及 100 多个哺乳动物对照物种；其中蝙蝠数据存在质量分组差异。后续结果展示需要明确哪些是高质量数据、哪些是对照物种、哪些是来自 Bat1K 或参考文献相关数据。
  5. 现有 embedding 图仍值得拆分重画，但要改成可解释的对照视图。 不建议继续只看一张混合散点图。更可行的是每个蛋白单独出图、每个蝙蝠物种分别与所有对照物种比较，并清楚标注颜色和符号代表的物种、蛋白或 control family。这样更容易观察某些蝙蝠物种是否在特定蛋白上形成离群或聚集趋势。
  6. BAT 项目成果展示应改为网页，而不是 PPT。 会议明确后续给老师看的材料优先做成 GitHub Pages/HTML 页面，把建树结果、embedding 图、物种对照和候选基因解释放在可点击的网页中。网页比 PPT 更适合承载多图、多物种、多层级结果，也更符合当前项目网站的展示方式。

可行方向

  1. 严格复现参考文献的比较基因组流程。 输入是已注释的蝙蝠和哺乳动物基因组/蛋白数据；方法是按参考文献流程做同源基因识别、物种树构建、分支变化分析和功能富集。输出应包括可复核的物种树、候选基因列表、富集通路和关键图。验证方式是看是否能复现参考文献中免疫相关基因在蝙蝠分支上的富集趋势。
  2. 重构 embedding 可视化。 输入是已有 ESM/Evo embedding 和候选免疫基因/蛋白清单；方法是按单蛋白、单蝙蝠物种、蝙蝠分组和对照物种分别出图。输出是一组可点击的散点图或图表页面。验证方式是每张图都能说明颜色、形状、物种和蛋白含义，并能回答是否存在蝙蝠离群或聚集趋势。
  3. 收敛候选免疫基因/蛋白解释链条。 输入是已知免疫、病毒进入、炎症调控或病毒耐受相关候选基因；方法是比较不同物种中的序列差异、蛋白片段变化、embedding 距离和功能注释。输出是“候选基因-功能-蝙蝠差异-证据图”的小表。验证方式是每个候选都能落到一个可解释的生物学问题，而不是只呈现抽象点云。
  4. 建立 BAT 项目 HTML 展示仓库。 输入是建树结果、embedding 图、物种清单和候选基因解释；方法是在 GitHub 上建立 BAT/BET 展示仓库，并用 GitHub Pages 发布。输出是一个可给老师直接浏览的网页。验证方式是网页能按项目问题、数据来源、分析流程、关键图和下一步组织内容。
