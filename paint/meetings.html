完整记录

核心结论

  1. 直接 embedding 全部物种和候选基因，暂时不能支撑明确的蝙蝠特异性结论。 当前结果中，免疫相关基因在不同物种里本来就可能和其他基因分开；这种分离更像功能类别差异，而不一定是蝙蝠相对其他哺乳动物的特异性。把大量蝙蝠、对照物种和多个蛋白混在同一张图里，也会掩盖单一物种或单一蛋白的局部趋势。
  2. 参考文献中的进化树/同源基因分支分析仍是更稳妥的主线。 会议讨论了先识别同源基因、构建物种进化树，再沿祖先到子节点的分支观察基因或蛋白变化，并对突变较集中的基因做功能注释的路线。该流程更接近比较基因组领域可接受的分析范式，也更容易把结果连接到免疫通路和病毒耐受机制。
  3. 当前复现效果不佳，主要瓶颈可能在同源基因规模、建树规模和流程简化造成的误差积累。 已尝试的小规模树、较少同源基因和简化比对流程，未能稳定复现参考文献中免疫相关基因富集的效果。后续不应急于基于小规模结果下结论，而应优先把参考流程做完整、做大、做可复核。
  4. 数据优势集中在蝙蝠物种和高质量注释子集上，需要先把数据边界讲清楚。 会议中提到已下载约 28 个有基因/蛋白注释的蝙蝠物种，以及 100 多个哺乳动物对照物种；其中蝙蝠数据存在质量分组差异。后续结果展示需要明确哪些是高质量数据、哪些是对照物种、哪些是来自 Bat1K 或参考文献相关数据。
  5. 现有 embedding 图仍值得拆分重画，但要改成可解释的对照视图。 不建议继续只看一张混合散点图。更可行的是每个蛋白单独出图、每个蝙蝠物种分别与所有对照物种比较，并清楚标注颜色和符号代表的物种、蛋白或 control family。这样更容易观察某些蝙蝠物种是否在特定蛋白上形成离群或聚集趋势。

完整记录

核心结论

  1. 论文核心不是单纯提出一个新生成器，而是把 drift 定义在 decoder feature space 中。 该方法先训练或使用 VAE encoder/decoder，再训练条件 generator 输出 latent z，最后通过 frozen decoder 解码成分子。关键在于 drift 不是直接在外部 feature 或原始 latent 上定义，而是通过 decoder hidden features 与分子结构生成过程耦合。
  2. decoder-coupled drift 的价值在于提供 pullback gradient。 drift vector 不直接更新分子结构，而是经过固定 decoder 的可微映射被拉回 latent space，从而更新 generator 参数。这样优化方向与 decoder 最终产生分子的方式保持一致，解释了为什么直接在 detached feature、外部 feature 或普通 latent space 中做 drift 效果可能较弱。
  3. 外部 feature/proxy metric 的效果可能不如 decoder 内部特征。 会议中提到，如果用外部编码器、分类器、打分器或手工 proxy metric 构造回归任务，优化目标可能与最终分子解码过程脱节。decoder feature 更接近实际生成机制，因此更适合作为 drift 和局部度量的空间。
  4. 多样性项用于缓解 collapse。 如果只按目标性质或 drift 聚合生成，模型可能收缩到少数性能不错但重复的分子。论文中的 z-diversity 或类似 repulsion/hinge 项用于把 latent samples 拉开，从而提升唯一性和多样性。
  5. 低推理成本来自两阶段构造。 由于 decoder frozen，generator 可单步输出 latent z，再直接 decode 成分子；推理时不需要 denoising、ODE solver 或 iterative refinement。这是该方法相较部分 diffusion/flow 方法的重要工程优势。
  6. 该论文更适合作为机制参考，而不是直接照搬。 会议中对数学推导和实验 protocol 进行了初步拆解，但部分术语和公式仍需回看原文确认。对团队而言，更有价值的是借鉴“生成空间应与解码器/最终结构空间耦合”的思想。

完整记录

核心结论

  1. 单步逐步搜索不适合级联反应规划。 如果每一步都独立生成多个候选反应类型，很容易出现第三步 output 无法作为第四步 input、上下步条件不兼容等问题。对项目推进而言，不能把任务简单拆成“每一步预测一个反应”，而要把整条路径作为对象评估。
  2. 级联规划必须具有全局路径视角。 会议中反复强调，模型需要在反应开始时就看到整条路线大概有多少步、哪些步骤可能合并、哪些条件需要统一。否则模型只能局部判断当前 action 是否合理，无法判断其对后续步骤的影响。
  3. 当前数据规模不足以支撑直接训练一个强级联规划模型。 只依赖现有几千条级联数据去训练或微调，容易得到不可靠的“骨架”或路线。后续应避免把主线押在从小数据中直接学出完整规划能力上。
  4. 小模型微调和蒸馏路线需要谨慎。 讨论中提到 LLaMA 3.1、小模型控制、on-policy distillation 等方向，但大家对“小模型能否复制顶级大模型输出”和“微调用什么任务才有意义”持保留态度。短期更现实的是借助强 API 或现成模型做评估与选择。
  5. 更可行的方案是“已有路线评估 + 级联修饰”。 让已有 planner 或组内模型先生成 top 100 路线，再判断哪些路线可改造成级联，哪些步骤可缩短，哪些条件可通过更换催化剂、酶或反应条件进行合并。这比从零生成级联路线更稳，也更容易验证。
  6. 不建议强行向 auto planner 注入高权重人工候选。 如果不给人工候选很高权重，planner 可能选不到；如果强行给高权重，又可能破坏原有规划逻辑。因此这条路更像临时补丁，不适合作为主线方法。

可行方向

  1. 候选路线级联可行性评估器。 输入是已有 planner 给出的多条候选合成路线；模型判断每条路线是否可能改造成级联路线，输出可级联概率、关键限制步骤和不兼容原因。验证方式是用人工标注或 cascade test set 检查排序与判断准确性。
  2. 路线缩短与节点合并。 输入是一条多步路线及每步反应类型、底物/产物、条件；模型识别可合并的相邻步骤或节点，输出缩短后的路线和修改理由。验证方式是检查合并后上下步物质连接、条件兼容性和化学合理性。
  3. 条件统一与规避建议。 输入是路线中各步温度、催化剂、酶、溶剂等条件；模型判断哪些步骤条件冲突、哪些条件可替换或统一，输出建议的条件调整方案。验证方式是由有机/催化背景成员复核。
  4. 强模型辅助的候选选择。 输入为每一步或每条路线的候选集合；LLM 根据全局路径上下文选择较优候选，而不是只看局部反应。验证方式是比较有无全局上下文时的路线连贯性和可执行性。
